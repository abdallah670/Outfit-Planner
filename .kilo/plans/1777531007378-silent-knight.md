## 2. Gap Analysis

### 2.1 URL Mismatches (Critical Fixes)

`SocialDataSource` must call:
- Polls → `${base}/api/polls` (not `/Polls`)
- Trending → `${base}/api/trending/outfits` (not `/Trending`)
- Feed posts (list, get, delete) → `${base}/api/feed`
- Reactions & comments → `${base}/api/feed/{id}/heart`, `/comments`
- Outfit posts → `${base}/api/outfitposts` (create, update, delete – currently not in Social layer)

### 2.2 DTO Shape Mismatches

**Trending:** Backend returns `TrendingOutfitDto` (from `TrendingOutfitDto.cs`):
```cs
{ Id, Name, Description, ImageUrl?, UserId, UserName, VoteCount, CommentCount, TrendingScore, CreatedAt }
```
Frontend `TrendingOutfit` expects similar fields but may need alignment:
- Map `Id` → `id`
- `Name` → `title` or `userName`? (design decision: `Name` likely outfit name)
- `ImageUrl` → `imageUrl`
- `VoteCount` → `likes` (or `voteCount`)
- `CommentCount` → `comments`
- `TrendingScore` → `trendingScore`
- Keep `userId`, `userName`, `userAvatar` (if available), `voteId` (not present – treat as optional)

**Feed Posts:** Backend `FeedPostDto` has `Id, UserId, UserName, UserAvatarUrl, PostType, OutfitId, Outfit?, PollId, Poll?, Caption, Visibility, LikeCount, CommentCount, UserReaction, CreatedAt`. Frontend `FeedPost` matches after date conversion.

**Polls:** Already match (`ValidationPollDto` ↔ `ValidationPoll`).

### 2.3 Missing Features

| Feature | Backend | Frontend |
|---|---|---|
| Follow/Unfollow button on profile | ✅ | ❌ |
| Followers/Following lists UI | ✅ | ❌ |
| Create Outfit Post (share to feed) | ✅ | ❌ |
| Edit/Delete Outfit Post | ✅ | ❌ |
| Like comment on vote comments | ❌ (no endpoint) | ❌ Effect exists |

### 2.4 Clean Architecture Violations

- `CreatePollComponent` uses direct HttpClient for image upload (`/poll-image-upload/upload`). Must move to `SocialRepository`.

## 3. Implementation Strategy

**Approach:** Merge `Feed*` classes into `Social*` by moving and combining code. Eliminate `FeedRepository`, `FeedDataSource`, `FeedUseCases`, and `feed` NgRx slice. All social operations (feed posts, polls, trending, follow, outfit posts) will go through `SocialRepository` and `social` NgRx slice.

**Steps:**
1. Extend `SocialDataSource` with all feed-related methods (`getFeedPosts`, `getPostById`, `deletePost`, `addReaction`, `removeReaction`, `getComments`, `addComment`, `deleteComment`, `createOutfitPost`).
2. Update `SocialRepository` interface to include feed methods.
3. Implement them in `SocialRepositoryImpl` by delegating to `SocialDataSource`.
4. Add feed use cases to `SocialUseCases`.
5. Migrate NgRx `feed` actions/effects/reducer/selectors into `social` slice:
   - Merge `FeedActions` into `SocialActions` (or keep separate but dispatch to same use cases).
   - Merge `feed.reducer` logic into `social.reducer` (add feed state fields).
   - Merge `feed.effects` into `social.effects`.
   - Add feed selectors to `social.selectors`.
6. Update components:
   - `CommunityFeedComponent` → use `SocialActions` instead of `FeedActions`.
   - Update all dependencies to import from social state.
7. Remove `Feed*` files (datasource, repository, usecases, state files) OR mark deprecated.
8. Update `app.config.ts`: Remove `FeedEffects`, `FeedRepositoryImpl` provider; keep only `SocialEffects`, `SocialRepositoryImpl` (which now handles feed).
9. Fix URLs in `SocialDataSource`.
10. Add Follow feature to Social layer (new domain entities, repository methods, actions, reducer, effects).
11. Create `uploadPollImage` in `SocialRepository` and refactor `CreatePollComponent`.
12. Handle trending DTO mapping (frontend adapts to backend `TrendingOutfitDto`).
13. Add "Share to Feed" outfit post creation via `SocialRepository`.
14. Implement follow UI on Profile page.

## 4. Detailed Implementation Steps

### Phase 0 – Backend: Add GetPublicProfile Endpoint (2–3 hrs)

**Goal:** Enable frontend to fetch basic profile info for any user (needed for viewing other users' profiles and follow UI).

1. **Create/Update DTO**: Ensure `UserProfileDto` (or new `PublicUserProfileDto`) contains public fields:
   - `Id`, `Name`, `ProfilePictureUrl`, `WardrobeItemCount`, `OutfitCount`, `TotalWears`
   - Sensitive fields (`Email`, `Preferences`, `StyleProfile` details) may be excluded for public view.

2. **Create Query & Handler**: `GetPublicUserProfileQuery` and `GetPublicUserProfileQueryHandler`:
   ```cs
   public class GetPublicUserProfileQuery : IRequest<UserProfileDto?> {
       public string UserId { get; set; } = string.Empty;
   }
   ```
   Handler fetches user by ID (include Wardrobe/Outfit counts) and maps to DTO. Returns null if not found.

3. **Add Endpoint** in `UserController.cs`:
   ```cs
   [HttpGet("users/{userId}/profile")]
   public async Task<ActionResult<UserProfileDto?>> GetPublicProfile(string userId) {
       var query = new GetPublicUserProfileQuery { UserId = userId };
       var profile = await _mediator.Send(query);
       if (profile == null) return NotFound();
       return Ok(profile);
   }
   ```

 4. **Integration Test**: Verify endpoint returns 200 with public fields.

## 7. Open Questions

1. **Trending entity field mapping:** The backend DTO `TrendingOutfitDto.Name` – should that map to `TrendingOutfit.title` (outfit name) or `userName`? Based on domain, `Name` likely refers to the outfit's name or a trend name. If the UI expects to show user avatar/name, we need `UserName` field which exists. So mapping: `Name` → `title` (shows outfit/trend name), `UserName` → `userName` (shows user who posted). This is acceptable.
2. **Poll image upload endpoint** (`/poll-image-upload/upload`) – stays separate; fine to call from `SocialDataSource` as it's still a data source method.
3. **Follow on profile:** Should clicking a user's name on profile page (own profile) allow following? Not applicable. Only when viewing another user's profile. Need a way to navigate to other profiles – perhaps from trending items (which have `userId`). If trending doesn't provide `userId`, follow may need to be a separate "discover users" page, or we show follow on profile when a `userId` route param is provided (e.g., from search or later integration).
4. **Feed post creation:** Use `POST /api/outfitposts`. The response should return a `FeedPostDto` that can be prepended to state as `FeedPost` entity.
