## User Interface Choice: SweetAlert2

The user has explicitly requested to use SweetAlert2 instead of Angular Material Dialog for the modal dialogs. This aligns with the existing project pattern where SweetAlert2 is widely used for alerts and confirmations. The modals will be implemented using SweetAlert2's `Swal.fire()` with custom HTML content.

### SweetAlert2 Modal Implementation Strategy

Since SweetAlert2 expects a configuration object with HTML strings (not Angular component templates), we need a different approach for complex content:

#### Option A: Inline HTML Templates (Recommended)
- Use `Swal.fire({ html: '<div id="modal-container"></div>', ... })`
- Dynamically render Angular component into the container using `ViewContainerRef` and `ComponentFactoryResolver` or `ViewChild`+`ComponentRef`
- Manage component lifecycle manually (create before show, destroy on close)
- Pros: Full Angular component capabilities (forms, change detection, services)
- Cons: More boilerplate, manual component management

#### Option B: Pure HTML+DOM Manipulation (Simpler but less Angular)
- Build HTML strings for comment/voter lists
- Use event delegation on container for click handlers
- Manage state in the parent component (PublicProfileComponent)
- Pros: Simpler, no dynamic component creation
- Cons: Not idiomatic Angular, harder to maintain, lose Angular data binding

**Decision: Use Option A** - create component but render into SweetAlert container dynamically.

#### Implementation Steps:
1. Create `CommentsModalComponent` as standalone (no entry in any module)
2. Create `VotersModalComponent` similarly
3. In `public-profile.component.ts`:
   - Inject `ViewContainerRef` and `ComponentFactoryResolver` (or use `ViewContainerRef.createComponent`)
   - Define methods `openCommentsModal(postId)` and `openVotersModal(optionId, pollId)`
   - Inside these methods:
     a. Create component instance dynamically
     b. Configure with input data (postId/optionId/pollId)
     c. Call `Swal.fire({ html: container, width: ..., showCloseButton: true, ... })`
     d. Handle SweetAlert close to destroy component
4. For confirmations (delete comment), use `Swal.fire({ icon: 'warning', ... })` built-in
5. For success/error messages after actions, use simple `Swal.fire({ icon: 'success', timer: 2000, ... })`

### Technical Considerations for SweetAlert Dynamic Components
- Use Angular's `ViewContainerRef.createComponent()` (Angular 14+) for dynamic component creation
- The container element for SweetAlert must be created programmatically
- Example approach:
```typescript
const container = document.createElement('div');
const componentRef = this.viewContainerRef.createComponent(CommentsModalComponent, { 
  injector: this.injector 
});
(componentRef.instance as CommentsModalComponent).postId = postId;
container.appendChild(componentRef.location.nativeElement);
Swal.fire({
  html: container,
  width: 600,
  showCloseButton: true,
  backdrop: true,
}).then((result) => {
  componentRef.destroy();
});
```
- Need to handle change detection: component runs in Angular zone automatically if created via ViewContainerRef

### Styling
- SweetAlert2 uses its own styling; modal content will have standard Angular component styling
- Set appropriate width (e.g., 600px for comments, 500px for voters)
- Use `customClass` option if needed

### Risks / Mitigation
- Dynamic component lifecycle: ensure destroy on close → use `willClose` or `finally`
- Memory leaks: explicitly destroy componentRef
- Change detection: ViewContainerRef handles automatically
- Container element must be attached to DOM before Swal opens; append to body or use Swal's pre-existing container
